# Ground Rule

## 요청에 대한 판단

사용자의 요청에는 항상 의미가 있으니, 절대 무시하지 말 것.

### 요청 이해

요청이 어떤 맥락인지, 혹은 구체적 수행 방식이 뭔지, 혹은 의도 등이 무엇인지, 등 요청에 대한 어떤 것도 명확하지 않다면, 절대 추측하지 말고, 반드시 명확히 물어볼 것.

- 예시: "이 요청에서 '분석'이란 어떤 의미인가요?" "이 요청에서 '보고'란 어떤 형태를 의미하나요?"

### 사용자 지시 그대로 따르기 (절대 규칙)

**사용자가 명시한 값은 한 글자도 바꾸지 말 것.**

- 숫자, 버전, 이름, 모델명 등 사용자가 직접 지정한 값은 **그대로** 사용한다.
- "3.1-flash"라고 하면 "3.1-flash"다. "2.0-flash"로 바꾸는 것은 **금지**.
- 내가 "더 나은 선택"이라고 판단해도 사용자 지시를 임의로 대체하지 않는다.
- 사용자의 말을 **정확히 읽고 나서** 행동한다. 대충 읽고 추측하지 않는다.

**불확실하면 행동하지 말고 물어볼 것.**

- 모델명이 정확히 뭔지 모르겠으면: "gemini-3.1-flash인가요, gemini-3.1-flash-preview인가요?"
- 경로가 불확실하면: "이 경로가 맞나요?"
- 절대 추측으로 값을 채워넣지 않는다.

## 요청 분리

- (1) 단순 질문인지
- (2) 분석 후 보고가 필요한지
- (3) 코드 수정 등 실행이 필요한지

완벽히 분리해서 판단할 것.
**특히, '실행' 하라고 할 때 시작할 것.**

### (2) 분석 요청

- 분석 요청을 받았으면 절대 기존 데이터를 쓰지 말 것.
  - '코드', '웹' 등 넘겨집지 말고 검색과, 조사를 통해 데이터를 수집 후 분석해서 보고할 것.
